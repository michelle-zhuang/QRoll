## Anti-Patterns Verdict

**Does this look AI-generated?** Yes, unmistakably. The single biggest tell is **card soup**: every section on every page is wrapped in the identical `AstroCard` component (rounded-3xl, same shadow, same border, same padding). Nine cards on the dashboard alone. No differentiation in elevation, color, or density between stat cards, chart cards, list cards, and the matrix card. A human designer would never give a stat counter and a full-width chart grid the same visual container treatment.

**Deterministic scan** (`detect.mjs`): 16 findings across 2 files.
- **1× overused-font**: Inter font in `Welcome.astro` (dead code — Astro starter component not used anywhere).
- **14× design-system-color**: 13 hardcoded colors in `Welcome.astro` (dead code) + 1 undocumented `hsl(0 78% 78%)` in `DashboardCharts.tsx` (live code — used for the "absent" bar color).
- **1× design-system-radius**: `border-radius: 16px` in `Welcome.astro` (dead code).

The detector confirms the user's instinct: the real design system is thin. Almost all findings are in dead starter code, but the one live finding (undocumented chart color) reveals that even active code drifts from the palette.

## Overall Impression

The "Playful Dream" color palette is genuinely nice — the lavender-tinted card shadows and pastel accents create a warm, friendly identity. The navbar is the one place with polished Framer Motion animation (layoutId tab indicators, AnimatePresence dropdowns). The login page's gradient blobs show design ambition.

**The single biggest opportunity**: Break the card monoculture. The dashboard should feel like a structured command center with clear visual hierarchy — stat strips, elevated feature sections, recessed data tables — not a uniform grid of identical white boxes.

## What's Working

1. **The color palette is cohesive and distinctive.** The lavender-tinted box shadows (`rgba(228,193,249,0.12)`) give cards a branded, warm glow that differs from generic gray shadows. The Playful Dream palette carries through login blobs, navbar avatar gradients, and accent colors.

2. **The navbar has real motion craft.** Framer Motion `layoutId` for the active tab underline and `AnimatePresence` for the dropdown/mobile drawer feel polished and intentional. This is the quality bar the rest of the app should match.

3. **Pill-shaped buttons are modern and touch-friendly.** `rounded-full` with `active:scale-[0.98]` micro-interaction gives buttons a physical feel. The variant system (default/destructive/outline/secondary/ghost/link) is well-structured.

## Priority Issues

### [P0] Card Soup — Flat Visual Hierarchy
**What**: Every section on every page uses the identical `AstroCard` component. The dashboard has 9 cards. The roster page has 2 cards. The new-event form is 1 card. There is zero visual differentiation between a tiny stat counter and a full-width attendance matrix.

**Why it matters**: When everything has the same container treatment, nothing stands out. The user's eye has no visual anchor. The stat "75% on-time rate" and the attendance matrix grid carry equal visual weight, which means neither commands attention.

**Fix**: Introduce 3 elevation tiers: (1) flush/borderless for stat strips and inline data, (2) standard card for primary content blocks, (3) elevated/featured for the most important section. Remove cards from stat counters entirely — use borderless inline metrics with subtle dividers. Give the attendance matrix a recessed `bg-muted/30` treatment instead of a floating card.

**Suggested command**: `/impeccable layout dashboard.astro`

### [P0] Zero Motion Outside the Navbar
**What**: The navbar has polished Framer Motion animations, but every other page is completely static. Cards don't animate in. Charts pop into existence. Lists have no entrance. Page transitions are instant cuts.

**Why it matters**: The contrast between the animated navbar and the static content creates a jarring quality gap. The app feels like two products stitched together — a polished navigation shell around a static prototype.

**Fix**: Add staggered entrance animations to dashboard stat cards (fade-up with 50ms stagger). Add a subtle scale-in for the attendance matrix. Charts should fade in after data loads. Page content should have a coordinated entrance, not an orchestrated spectacle — 150-250ms state-conveying motion per the product register.

**Suggested command**: `/impeccable animate dashboard.astro`

### [P1] Double `<main>` Nesting (Semantic HTML Violation)
**What**: `Layout.astro` wraps `<slot>` in `<main>`, and every page also wraps its content in `<main>`. This produces `<main><main>` — a violation of the HTML spec (only one `<main>` per document).

**Why it matters**: Screen readers use `<main>` as a landmark. Two nested `<main>` elements confuse assistive technology and break skip-to-content navigation.

**Fix**: Remove the `<main>` from `Layout.astro` (keep only the page-level `<main>`) or remove the page-level `<main>` tags and keep the layout one.

**Suggested command**: `/impeccable audit src/layouts/Layout.astro`

### [P1] Roster Page is Visually Cramped
**What**: Each roster member row packs name, email-update form with input+save button, an account-link dropdown, and a delete button into a single `grid-cols-[1fr_1.2fr_1.2fr_auto]` row. On mobile it stacks into a tall, confusing block.

**Why it matters**: Admins managing 20+ members will struggle to scan and edit this dense layout. The inline forms make every row feel editable all the time, increasing cognitive load.

**Fix**: Show read-only member rows by default. Move editing into an expand/modal interaction triggered by clicking the row. On mobile, use a slide-out sheet per member.

**Suggested command**: `/impeccable layout admin/roster.astro`

### [P1] Narrow Typography Scale + Inconsistent Spacing
**What**: Only 4 text sizes across the entire UI (`text-3xl`, `text-lg`, `text-sm`, `text-xs`). Spacing alternates between `gap-4`, `gap-6`, `mb-6`, `mb-8` with no consistent rhythm. Card padding varies from `p-5` to `p-8`.

**Why it matters**: The narrow type scale flattens visual hierarchy — page titles and card titles don't feel different enough. Inconsistent spacing creates a "slightly off" feeling that accumulates across the interface.

**Fix**: Define a 6-step type scale (display/headline/title/body/label/caption) and a spacing scale (4/8/12/16/24/32px) as design tokens. Standardize card padding to one value (24px).

**Suggested command**: `/impeccable typeset src/`

## Persona Red Flags

**Casey (Distracted Mobile User)**: The check-in page is Casey's primary flow — scanning a QR on a phone at the studio. The "Submit attendance" button is the most important tap target in the app. Currently it's well-sized (`h-14`) and at the bottom of the card — good. But the page loads with no motion feedback, and if GPS is slow there's no loading indicator. Casey will wonder if anything is happening.

**Alex (Impatient Power User / Admin)**: The admin managing a 30-person dance roster has no search, no bulk actions, no keyboard shortcuts. Adding 30 members means 30 individual form submissions. The events table has no filters. Alex will tolerate this for a small team but it won't scale.

**Jordan (First-Timer / New Attendee)**: Scanning the QR code for the first time, Jordan lands on `/checkin/[token]` and sees "Your account is not linked to a roster entry yet. Please ask an admin to link you." — a dead end with no help or next step. No guidance on what "linked" means or how to resolve it.

## Minor Observations

- **Dead code**: `Welcome.astro` is the default Astro starter component with hardcoded off-brand colors. Should be deleted.
- **No dark mode**: `@custom-variant dark` is configured in CSS but no `.dark` color values exist. Dark mode toggle would render broken.
- **Redundant auth**: Layout fetches session+profile for Navbar, then every page re-fetches the same data (2× Supabase round-trips per page load).
- **No responsive chart sizing**: Charts have fixed heights (`h-[320px]`, `h-[420px]`) that may overflow on small screens.
- **Mixed icon systems**: Navbar uses Lucide React, login uses `react-icons/fc`, all other pages use raw inline SVGs. Should standardize on Lucide.
